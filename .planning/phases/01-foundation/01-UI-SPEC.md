## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Helper text spacing, chip padding, tight form gaps |
| md | 16px | Default form spacing, card padding, screen gutters |
| lg | 24px | Section padding, stacked card gaps |
| xl | 32px | Major screen gaps, banner offset from content |
| 2xl | 48px | Hero/status block spacing |
| 3xl | 64px | Top-level page spacing on taller screens |

Exceptions: minimum tap target 44px tall for all tappable controls; primary buttons 48px tall; segmented auth switch options 44px tall.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.35 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Use exactly two weights across the phase: 400 and 600. Body copy stays at 16px to preserve legibility in iOS standalone mode. Labels and buttons use 14px/600. Household status and auth headings use 20px. One display size is allowed for the signed-in household status block and success states only.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F7F3EB` | App background, auth surfaces, large page areas |
| Secondary (30%) | `#E7DED0` | Cards, segmented control track, banner shell, detail panels |
| Accent (10%) | `#176B5D` | Primary submit button, active auth segment, `Copy code`, success badge, install banner emphasis |
| Destructive | `#B9382F` | Inline destructive text, destructive confirmations only |

Accent reserved for: auth submit button, active `Log in`/`Sign up` state, `Copy code`, join/create success chips, and the install-guidance icon or emphasized instruction text. Do not use accent for both household choice cards; `Create household` and `Join with code` must remain equal-weight options.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Log in` |
| Empty state heading | `No household yet` |
| Empty state body | `Create a household or join with a code to start sharing expenses.` |
| Error state | Auth: `Email or password didn't match. Try again.` Join: `That code is invalid, expired, or already used. Ask for a new code and try again.` |
| Destructive confirmation | None in Phase 1. Logout is non-destructive and should execute immediately from a secondary action. |

Additional copy rules:
- Auth screen includes one short framing sentence above the form. Use practical onboarding copy, not marketing language.
- Install guidance copy is literal and action-led: `Tap Share, then Add to Home Screen.`
- Success states should be brief and confidence-building. Cap success body copy at one sentence.

## Surface Inventory

This phase defines the first mobile surfaces for the app:

| Surface | Contract |
|---------|----------|
| Combined auth screen | Single screen with segmented `Log in` / `Sign up` switch. Default to `Log in`. Email + password only. Password hidden by default with show/hide toggle. Inline errors stay on-screen. |
| Household choice screen | Two vertically stacked cards or buttons: `Create household` and `Join with code`. Equal visual weight. No default primary path. |
| Create household screen | Minimal form with one field: household name. No explainer step. One full-width submit button. |
| Join household screen | Dedicated code-entry screen with one standard text input, paste support, submit-time validation only, inline retryable error, and clear secondary action to switch to create flow. |
| Join confirmation screen | Show household name only, then require explicit confirmation before joining. |
| Join success state | Brief success state, then auto-continue into signed-in shell. |
| Signed-in shell | Household-connected status surface. Show household identity, member status, and a primary reachable action: `View household details`. Do not preview future Quick Add features. |
| Household details screen | Single summary view combining household identity and member status. |
| Invite management screen | Lives under menu/settings, not the main shell. Show active code if present, expiry, `Copy code`, and joined status after use. |
| Install guidance banner | Non-modal banner/card shown only after successful auth in Safari, never on first paint, never in standalone mode. Dismissal snoozes rather than permanently hides. |

## Interaction Contract

- Keep all primary actions full-width and anchored within easy thumb reach on mobile screens.
- Prefer single-column layouts throughout Phase 1. Only use side-by-side layouts for the auth segmented switch itself.
- Auth and onboarding transitions should feel direct. Avoid multi-step explainers, coach marks, or feature tours.
- Join-code validation happens on submit only. Do not show live validation states while typing.
- Errors appear inline, directly tied to the field or form that failed. Do not use blocking modals for auth or invite failures.
- Success states may auto-advance after 1 to 2 seconds. Provide visible confirmation before transition.
- Invite management is intentionally separated from the main shell to keep the signed-in home focused on connected status.
- The main shell must communicate "connected and ready" rather than "empty product". Use real household data blocks, not placeholders for future phases.
- Install guidance appears only after auth success on Safari. It should read like an instruction card, not a promotional upsell.

Source notes: interaction rules are primarily locked by `.planning/phases/01-foundation/01-CONTEXT.md` decisions D-01 through D-39, with scope and success criteria from `.planning/ROADMAP.md` and `.planning/REQUIREMENTS.md`.
