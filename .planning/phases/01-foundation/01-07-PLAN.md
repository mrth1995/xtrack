# Phase 01 Plan 07: UAT Gap Closure

## Objective

Close Phase 1 UAT gaps from `01-UAT.md`: fail fast on bad Supabase env values, expose logout in the signed-in shell, make invite lookup/reuse reliable through database RPCs, and add regression coverage for the fixed behavior.

## Gaps Addressed

- Test 1: Cold start crashes with `placeholder.supabase.co`
- Test 2: Migration/setup validation blocked by placeholder Supabase URL
- Test 5: No visible logout button
- Test 7: Valid invite code shows generic invalid/expired/used message
- Test 8: Invite management always replaces existing valid code
- Test 9: Household details/member list blocked by failed invite flow

## Tasks

### Task 1: Add Supabase Environment Guard

**Goal:** Invalid Supabase env values produce an actionable setup error before app code attempts network calls.

**Implementation:**

- Add a small shared helper for validating `PUBLIC_SUPABASE_URL` and `PUBLIC_SUPABASE_ANON_KEY`.
- Reject empty values, `placeholder.supabase.co`, and non-URL Supabase URL values.
- Use the helper in both browser and server Supabase client setup.
- Keep service-role keys out of client/server public client paths.
- Add tests for missing/placeholder env handling where feasible without leaking secrets.

**Acceptance criteria:**

- Starting the app with placeholder Supabase URL surfaces an actionable setup error.
- Real-looking Supabase URLs continue to create clients normally.
- `npm run check` and `npm test` pass.

### Task 2: Add Visible Logout Control

**Goal:** A signed-in user can discover and trigger logout from the app UI.

**Implementation:**

- Add a visible logout form/button to the signed-in shell, likely below the household/invite actions.
- Use the existing `/logout` endpoint with POST semantics.
- Keep the control mobile-friendly and visually secondary.
- Add or update shared-shell tests to assert the logout control exists.

**Acceptance criteria:**

- Signed-in home shows a logout control.
- Triggering logout routes through `/logout` and returns to `/auth`.
- Regression tests cover the visible control.

### Task 3: Move Invite Reuse Into a Database RPC

**Goal:** Invite management returns an existing active invite before creating a new one, without relying on an RLS-scoped client-side read that can fail silently.

**Implementation:**

- Add a migration with a `get_or_create_active_household_invite(p_household_id uuid)` `SECURITY DEFINER` function.
- Inside the function:
  - Verify `auth.uid()` is a member of `p_household_id`.
  - Return the newest active invite where `used_at IS NULL`, `revoked_at IS NULL`, and `expires_at > now()`.
  - Create a new invite only when no active invite exists.
- Update `getOrCreateActiveInviteCode` to call the RPC directly.
- Remove the direct `household_invites` read/fallthrough path from the TypeScript service.
- Update DB types and service tests for reuse and create cases.

**Acceptance criteria:**

- Loading invite settings repeatedly returns the same active code until it is used, revoked, or expired.
- No direct client-side invite-table read is required for reuse.
- Creation errors are surfaced instead of silently replacing codes.

### Task 4: Harden Invite Lookup Error Handling

**Goal:** A missing/unapplied lookup RPC is not presented as a user-entered invalid code.

**Implementation:**

- Update `lookupInviteCode` to distinguish known invite hints from infrastructure/RPC errors.
- Preserve the user-friendly invalid/expired/used copy for known invite states only.
- Surface a setup/deployment-oriented error for missing RPC/function errors.
- Add unit tests for known invite hints and unknown RPC errors.

**Acceptance criteria:**

- Valid-code lookup uses `lookup_household_invite`.
- Missing RPC errors are not collapsed into `invalid_code`.
- The join flow can show confirmation for a valid active invite after migrations are applied.

### Task 5: Add Verification Coverage and Docs

**Goal:** Manual UAT has a clear setup path and automated tests catch the invite regressions.

**Implementation:**

- Add structural tests asserting the new RPC names exist in migrations and DB types.
- Extend service tests for:
  - active invite reuse through RPC
  - create-on-no-active-invite through RPC
  - unknown lookup RPC error copy/hint
  - visible logout control
- Update README or ops docs with the post-fix migration/UAT sequence:
  - configure `.env`
  - apply migrations
  - restart dev server
  - retest invite creation/join

**Acceptance criteria:**

- `npm test` passes.
- `npm run check` passes.
- UAT retest path for tests 1, 2, 5, 7, 8, and 9 is documented.

## Verification

Run:

```bash
npm test
npm run check
npm run build
```

Manual retest:

- `$gsd-verify-work 1` after applying the new migration to the target Supabase database.

## Risks

- Hosted Supabase must receive the new migration before invite lookup/reuse can pass manual UAT.
- Local `.env` is gitignored, so automated tests can only validate guard behavior and docs, not the user's actual local credentials.
