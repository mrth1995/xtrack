  <how-to-verify>
**Setup:** Open http://localhost:5173/ in Chrome (mobile emulation) AND on an iPhone (Safari, if available). Log in with your test household account.

---

**Block A — Quick Add core (INPUT-01, INPUT-02, INPUT-03, INPUT-04, INPUT-05)**

1. After login the URL should be `/` and the screen should show: install banner (or hidden if standalone), gear icon top-right, big "0" amount display, 12-key numpad, 7 category tiles in 3-3-1 layout, "Today" header, and "View all history →" link.
   - **Pass criterion:** No dashboard, no menu, no extra screen between login and the numpad (INPUT-01).
2. Tap 5, 4, 0, 0, 0 on the numpad. The amount display should update in real time:
   `5 → 54 → 540 → 5.400 → 54.000`.
   - **Pass criterion:** dot separator, no currency symbol, instant feedback (INPUT-02, INPUT-03).
3. Confirm the 7 category tiles render with these exact emojis (D-13):
   Row 1: 🍜 Food, 🚗 Transport, 🎬 Entertainment.
   Row 2: 💡 Utilities, 🛍️ Shopping, ❤️ Health.
   Row 3: ✅ Others.
4. With "54.000" entered, tap Food. The note bottom sheet should slide up showing "🍜 Food · Rp 54.000". The amount display should reset to "0".
   - **Pass criterion:** save fires, numpad resets, sheet slides up (INPUT-05, D-04, D-07).

---

**Block B — Debounce + system-keyboard suppression (INPUT-06, D-05)**

5. Dismiss the note sheet via "Skip". Tap the numpad to enter another amount, then double-tap rapidly on "Transport". Only ONE Transport row should appear in the today list (after the sheet dismisses).
   - **Pass criterion:** 500ms debounce prevents duplicate (INPUT-06).
6. On iOS Safari (or mobile emulation), tap any numpad digit. The device system keyboard MUST NOT open.
   - **Pass criterion:** D-05 — buttons-only numpad, no `<input type="number">` anywhere.

---

**Block C — Note sheet + history (INPUT-14, D-08, D-09, D-16)**

7. Enter another amount, tap a category. In the bottom sheet, type "lunch" and tap "Save note". The sheet should close and the today-list row should now show "lunch" next to the category.
8. Save another expense, this time tap "Skip" instead of "Save note". The sheet closes; the row appears without a note. Reload the page — Skip should NOT re-prompt (D-09).
9. Tap "View all history →" at the bottom. URL navigates to /expenses (D-16).

---

**Block D — Edit + delete (INPUT-12, INPUT-13, D-06, D-17)**

10. On /expenses, expenses should be grouped under a date label (e.g. "26 Apr"). Tap any row.
    - **Pass criterion:** URL is `/expenses/{id}/edit` (D-06).
11. The edit form should show 4 editable fields: amount, category (3-3-1 grid with current category highlighted), note, date.
12. Change the amount to a new value (e.g. clear and type "75000"), pick a different category, edit/clear the note, change the date back one day. Tap "Save changes".
    - **Pass criterion:** redirect to /expenses; the row reflects the new values; date label moved to the previous day (INPUT-12, D-17).
13. Open the same expense again. Tap "Delete expense". Button text changes to "Tap again to confirm delete" with destructive outline. Tap again.
    - **Pass criterion:** redirect to /; the row is gone from both / today list AND /expenses history (INPUT-13 — soft delete).
14. (Optional but recommended) Open Supabase SQL editor and run `SELECT id, is_deleted FROM expenses WHERE id = '<deleted-id>'`. The row should still exist with `is_deleted = true` (NOT a hard delete).
    - **Codex HIGH — saveNote privilege escalation verification (Cycle 3 UPDATED):** In the Supabase SQL editor, run `SELECT id, household_id, created_by FROM expenses ORDER BY created_at DESC LIMIT 5`. Note the `id` of an expense from your household. Then confirm that the `saveNote` action on any `expense_id` you do NOT own (from a different `created_by`) returns a **404/error response** (the action now returns `fail(404)` when 0 rows are updated, NOT "silent success" — RLS blocks the update and the 0-rows defense fires). The client-side toast "Expense not found" should appear. Note: if a second test account is unavailable, document as "deferred to Phase 3 cross-account test". **The verifier should look for an error toast, not the absence of a note change — the old Cycle 1/2 UAT said "silently return success" which is now WRONG.**
    - **Codex HIGH — edit/delete RLS regression check:** Run `SELECT policyname FROM pg_policies WHERE tablename = 'expenses'` in the Supabase SQL editor. Confirm the output includes `expenses_creator_update` and `expenses_household_select`. If either is missing, the authorization boundary for saveEdit and deleteExpense is broken and must be fixed before Phase 2 is marked complete.

---

**Block E — Gear menu (D-15)**

15. Tap the ⚙ gear icon top-right. Dropdown shows: Invite member, Household details, Log out.
16. Tap "Invite member" → /settings/invite loads. Tap browser back to /. Tap gear → "Household details" → /household loads.
17. Tap gear → "Log out". You're signed out and redirected to /auth.

---

**Block F — Edge cases (Pitfall coverage)**

18. WIB boundary: Set your device clock to 06:00 WIB (or just verify with whatever time it is). Log a fresh expense at the current time. It MUST appear in the "Today" list immediately.
    - **Pass criterion:** WIB date window is correct (Pitfall 6).
19. Amount overflow: Try to enter "999.999.9990" (i.e. type 9 ten times). The 10th digit should be ignored.
    - **Pass criterion:** max 99.999.999 enforced (UI-SPEC interaction step 3).
20. Empty-amount tap: With amount display showing "0", tap a category. Nothing should happen — no save, no sheet.
    - **Pass criterion:** category tap is a no-op when amount is empty.
21. **Codex Cycles 2+3 HIGH — duplicate-save idempotency (fully implemented):** Log an expense normally. Immediately tap another category tile rapidly (double-tap or retry scenario). The today list should NOT show a duplicate row. The note sheet should open for the original saved expense.
    - Code inspection: `grep -c "'23505'" src/routes/(app)/+page.server.ts` returns `1` — the duplicate client_id Postgres UNIQUE violation is caught explicitly.
    - Code inspection: `grep -c "duplicate: true" src/routes/(app)/+page.server.ts` returns at least `1` — the recovered existing row is returned as a success response.
    - Code inspection: `grep -c "count === 0" src/routes/(app)/+page.server.ts` returns `1` — saveNote 0-rows defense (Codex review) is present.
    - Code inspection: `grep -c "fail(404" src/routes/(app)/+page.server.ts` returns `1` — saveNote returns fail(404) when targeting a stale/deleted/cross-user expense.
    - **Code inspection (Cycle 3 HIGH — idempotency-safe prepend):** `grep -c "todayExpenses.some" src/routes/(app)/+page.svelte` returns `1` — the use:enhance handler uses `.some((e) => e.id === inserted.id)` to guard the prepend, not the `duplicate: true` flag alone. This covers both the double-tap (already visible, no-op) and network-retry (not yet visible, prepend) cases.
    - **Unit test verification (Cycle 3 MEDIUM — deterministic 23505 path):** `npm run test -- tests/expenses/quick-add.test.ts --reporter=verbose 2>&1 | grep -c "23505\|duplicate"` returns at least `1` — confirms `quick-add.test.ts` directly exercises the `23505` server path (not just a rapid-tap simulation which may miss the duplicate path after client_id regeneration).
    - **Note on zero-amount validation (Codex Cycle 4 LOW):** Amount=0 returns early client-side in `onCategoryTap` (before any form submission or server call), so no error toast appears — this is correct behavior per Step 20. The error toast for actual server-side failures (non-zero amount, 500 DB error) is verified via `grep -c 'showError' src/routes/(app)/+page.svelte` returning at least `2` in the code inspection above.
    - **Pass criterion:** A duplicate client_id is now handled gracefully as an idempotent success at both the server layer (23505 recovery with `is_deleted=false` filter — Cycle 4 MEDIUM fix) and the client layer (idempotency-safe prepend via .some()). Phase 3 will add SQL-level `ON CONFLICT DO NOTHING`. Server-side validation failures (non-zero amounts that pass Zod but fail at DB layer) surface as error toast.

22. **Codex Cycle 3 MEDIUM — deterministic 23505 unit test:** Verify that `tests/expenses/quick-add.test.ts` contains a test that exercises the `23505` duplicate path directly (not via UI rapid-tap simulation). Run:
    `npm run test -- tests/expenses/quick-add.test.ts --reporter=verbose 2>&1 | grep -E "23505|duplicate.*client_id|returns.*duplicate.*true"`
    - **Pass criterion:** At least one test in the `saveExpense action` describe block mocks the Supabase `insert` to return `code: '23505'`, mocks the subsequent `select` to return the existing row, and asserts the action returns `{ success: true, duplicate: true, expense }`. If this test is absent, add it to `tests/expenses/quick-add.test.ts` during execution of P02 before the human UAT check.
